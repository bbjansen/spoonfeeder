# Coding Standards

This document defines the coding conventions and patterns for projects generated by spoonfeeder.

---

## Naming Conventions

### File Naming

| Element     | Case Style | Format                  | Example                      |
| ----------- | ---------- | ----------------------- | ---------------------------- |
| Module      | kebab-case | `<name>.module.ts`      | `user.module.ts`             |
| Controller  | kebab-case | `<name>.controller.ts`  | `user.controller.ts`         |
| Service     | kebab-case | `<name>.service.ts`     | `user.service.ts`            |
| Entity      | kebab-case | `<name>.entity.ts`      | `user.entity.ts`             |
| DTO         | kebab-case | `<name>.dto.ts`         | `create-user.dto.ts`         |
| Interface   | kebab-case | `<name>.interface.ts`   | `user-response.interface.ts` |
| Mapper      | kebab-case | `<name>.mapper.ts`      | `user.mapper.ts`             |
| Constants   | kebab-case | `<name>.constants.ts`   | `queue.constants.ts`         |
| Guard       | kebab-case | `<name>.guard.ts`       | `auth.guard.ts`              |
| Interceptor | kebab-case | `<name>.interceptor.ts` | `response.interceptor.ts`    |
| Filter      | kebab-case | `<name>.filter.ts`      | `http-exception.filter.ts`   |
| Pipe        | kebab-case | `<name>.pipe.ts`        | `validation.pipe.ts`         |
| Test        | kebab-case | `<name>.spec.ts`        | `user.service.spec.ts`       |

### Class/Type Naming

| Element     | Case Style | Suffix        | Example               |
| ----------- | ---------- | ------------- | --------------------- |
| Module      | PascalCase | `Module`      | `UserModule`          |
| Controller  | PascalCase | `Controller`  | `UserController`      |
| Service     | PascalCase | `Service`     | `UserService`         |
| Entity      | PascalCase | `Entity`      | `UserEntity`          |
| DTO         | PascalCase | `Dto`         | `CreateUserDto`       |
| Interface   | PascalCase | -             | `UserResponse`        |
| Mapper      | PascalCase | `Mapper`      | `UserMapper`          |
| Guard       | PascalCase | `Guard`       | `AuthGuard`           |
| Interceptor | PascalCase | `Interceptor` | `ResponseInterceptor` |
| Filter      | PascalCase | `Filter`      | `HttpExceptionFilter` |
| Enum        | PascalCase | -             | `UserStatus`          |

### Variable/Property Naming

| Element          | Case Style            | Example                   |
| ---------------- | --------------------- | ------------------------- |
| Class properties | camelCase             | `userName`, `createdAt`   |
| Constants        | SCREAMING_SNAKE_CASE  | `MAX_RETRY_COUNT`         |
| Enum values      | PascalCase            | `Active`, `Pending`       |
| Database columns | snake_case            | `user_name`, `created_at` |
| Database tables  | snake_case (singular) | `user`, `migration_task`  |

## Dependency Injection

### General Conventions

```typescript
// CORRECT: Constructor injection with private readonly
@Injectable()
export class UserService {
  constructor(
    private readonly userRepository: UserRepository,
    private readonly configService: ConfigService,
    private readonly logger: Logger,
  ) {}
}

// INCORRECT: Property injection
@Injectable()
export class UserService {
  @Inject() userRepository: UserRepository; // Don't do this
}
```

### Rules

1. **Constructor injection only** - Never use property injection
2. **Dependencies as `private readonly`** - Unless mutation is specifically required
3. **Maximum 3 business dependencies** - Excluding Logger and ConfigService
4. **Inject interfaces when available** - For loose coupling
5. **Order dependencies** by importance to the class

### Custom Providers

```typescript
// Use tokens for non-class dependencies
export const SAM_API_KEY = 'SAM_API_KEY';

@Module({
  providers: [
    {
      provide: SAM_API_KEY,
      useFactory: (configService: ConfigService) => configService.get('sam.apiKey'),
      inject: [ConfigService],
    },
    SamService,
  ],
})
export class SamModule {}

// Usage
@Injectable()
export class SamService {
  constructor(@Inject(SAM_API_KEY) private readonly apiKey: string) {}
}
```

## Modules

### Module Structure

```typescript
@Module({
  imports: [
    // External modules first
    HttpModule,
    TypeOrmModule.forFeature([UserEntity]),
    // Internal modules second
    SharedModule,
  ],
  controllers: [UserController],
  providers: [UserService, UserMapper],
  exports: [UserService], // Only export what other modules need
})
export class UserModule {}
```

### Rules

1. **One module per feature** - Group related functionality
2. **Explicit imports** - Import only what you need
3. **Minimal exports** - Export only public API
4. **No circular dependencies** - Use forwardRef() only as last resort

## Controllers

### Conventions

```typescript
@Controller('users')
@ApiTags('Users')
export class UserController {
  constructor(
    private readonly userService: UserService,
    private readonly userMapper: UserMapper,
  ) {}

  @Get()
  @ApiOperation({ summary: 'Get all users' })
  async getUsers(): Promise<UserResponseDto[]> {
    const users = await this.userService.findAll();
    return users.map((user) => this.userMapper.toResponse(user));
  }

  @Get(':id')
  @ApiOperation({ summary: 'Get user by ID' })
  async getUserById(@Param('id', ParseIntPipe) id: number): Promise<UserResponseDto> {
    const user = await this.userService.findById(id);
    return this.userMapper.toResponse(user);
  }

  @Post()
  @ApiOperation({ summary: 'Create a new user' })
  @HttpCode(HttpStatus.CREATED)
  async createUser(@Body() dto: CreateUserDto): Promise<UserResponseDto> {
    const user = await this.userService.create(dto);
    return this.userMapper.toResponse(user);
  }

  @Put(':id')
  @ApiOperation({ summary: 'Update user' })
  async updateUser(
    @Param('id', ParseIntPipe) id: number,
    @Body() dto: UpdateUserDto,
  ): Promise<UserResponseDto> {
    const user = await this.userService.update(id, dto);
    return this.userMapper.toResponse(user);
  }

  @Delete(':id')
  @ApiOperation({ summary: 'Delete user' })
  @HttpCode(HttpStatus.NO_CONTENT)
  async deleteUser(@Param('id', ParseIntPipe) id: number): Promise<void> {
    await this.userService.delete(id);
  }
}
```

### Method Naming

| HTTP Method | Action | Method Name         | Example         |
| ----------- | ------ | ------------------- | --------------- |
| GET         | List   | `get<Entity>s()`    | `getUsers()`    |
| GET         | Single | `get<Entity>ById()` | `getUserById()` |
| POST        | Create | `create<Entity>()`  | `createUser()`  |
| PUT/PATCH   | Update | `update<Entity>()`  | `updateUser()`  |
| DELETE      | Delete | `delete<Entity>()`  | `deleteUser()`  |

### Rules

1. **Controllers handle HTTP concerns only** - No business logic
2. **Validate input** - Use DTOs with class-validator
3. **Transform output** - Use mappers for response DTOs
4. **Set appropriate status codes** - Use `@HttpCode()` decorator
5. **Document with Swagger** - Use `@Api*` decorators

## Services

### Conventions

```typescript
@Injectable()
export class UserService {
  private readonly logger = new Logger(UserService.name);

  constructor(
    @InjectRepository(UserEntity)
    private readonly userRepository: Repository<UserEntity>,
  ) {}

  async findAll(): Promise<UserEntity[]> {
    return this.userRepository.find();
  }

  async findById(id: number): Promise<UserEntity> {
    const user = await this.userRepository.findOne({ where: { id } });
    if (!user) {
      throw new NotFoundException(`User with ID ${id} not found`);
    }
    return user;
  }

  async create(dto: CreateUserDto): Promise<UserEntity> {
    const user = this.userRepository.create(dto);
    return this.userRepository.save(user);
  }

  async update(id: number, dto: UpdateUserDto): Promise<UserEntity> {
    const user = await this.findById(id);
    Object.assign(user, dto);
    return this.userRepository.save(user);
  }

  async delete(id: number): Promise<void> {
    const result = await this.userRepository.delete(id);
    if (result.affected === 0) {
      throw new NotFoundException(`User with ID ${id} not found`);
    }
  }
}
```

### Rules

1. **Services contain business logic** - Core application logic lives here
2. **Services can aggregate** - Combine data from multiple repositories/services
3. **Throw appropriate exceptions** - Use NestJS built-in exceptions
4. **Keep services focused** - Split large services into smaller ones
5. **Inject Logger** - Use NestJS Logger for structured logging

## DTOs and Validation

### Request DTOs

```typescript
import { IsString, IsEmail, IsOptional, MinLength, MaxLength } from 'class-validator';
import { ApiProperty, ApiPropertyOptional } from '@nestjs/swagger';

export class CreateUserDto {
  @ApiProperty({ description: 'User full name', example: 'John Doe' })
  @IsString()
  @MinLength(2)
  @MaxLength(100)
  name: string;

  @ApiProperty({ description: 'User email address', example: 'john@example.com' })
  @IsEmail()
  email: string;

  @ApiPropertyOptional({ description: 'User phone number' })
  @IsOptional()
  @IsString()
  phone?: string;
}
```

### Response DTOs

```typescript
import { ApiProperty } from '@nestjs/swagger';

export class UserResponseDto {
  @ApiProperty({ description: 'User ID', example: 1 })
  id: number;

  @ApiProperty({ description: 'User full name', example: 'John Doe' })
  name: string;

  @ApiProperty({ description: 'User email address', example: 'john@example.com' })
  email: string;

  @ApiProperty({ description: 'Creation timestamp' })
  createdAt: Date;
}
```

### Rules

1. **Use class-validator** - For request DTO validation
2. **Use class-transformer** - For transformation when needed
3. **Document with Swagger** - Use `@ApiProperty` decorators
4. **Separate request/response** - Different DTOs for different purposes
5. **No business logic** - DTOs are data containers only

### Validation Pipe Setup

```typescript
// main.ts
app.useGlobalPipes(
  new ValidationPipe({
    whitelist: true, // Strip non-decorated properties
    forbidNonWhitelisted: true, // Throw on extra properties
    transform: true, // Auto-transform to DTO class
    transformOptions: {
      enableImplicitConversion: true,
    },
  }),
);
```

## Entities

### Conventions

```typescript
import {
  Entity,
  PrimaryGeneratedColumn,
  Column,
  CreateDateColumn,
  UpdateDateColumn,
  Index,
} from 'typeorm';

@Entity('user')
export class UserEntity {
  @PrimaryGeneratedColumn()
  id: number;

  @Column({ length: 100 })
  name: string;

  @Index()
  @Column({ unique: true })
  email: string;

  @Column({ type: 'enum', enum: UserStatus, default: UserStatus.Active })
  status: UserStatus;

  @CreateDateColumn({ name: 'created_at' })
  createdAt: Date;

  @UpdateDateColumn({ name: 'updated_at' })
  updatedAt: Date;
}
```

### Rules

1. **One entity per file** - `<name>.entity.ts`
2. **Use snake_case** - For table and column names
3. **Singular table names** - `user` not `users`
4. **Include timestamps** - `created_at`, `updated_at`
5. **Index foreign keys** - And frequently queried columns
6. **Use enums** - For fixed value sets

## Mappers

### When to Use Mappers

- **Required**: When entity structure differs significantly from response DTO
- **Required**: When combining data from multiple sources
- **Optional**: When entity can be directly mapped to response

### Mapper Structure

```typescript
@Injectable()
export class UserMapper {
  /**
   * Maps a UserEntity to UserResponseDto
   */
  toResponse(entity: UserEntity): UserResponseDto {
    return {
      id: entity.id,
      name: entity.name,
      email: entity.email,
      createdAt: entity.createdAt,
    };
  }

  /**
   * Maps CreateUserDto to partial entity for creation
   */
  toEntity(dto: CreateUserDto): Partial<UserEntity> {
    return {
      name: dto.name,
      email: dto.email,
    };
  }
}
```

### Rules

1. **No business logic** - Pure data transformation only
2. **One mapper per entity** - When needed
3. **Stateless** - No internal state
4. **Single responsibility** - One input type to one output type per method

## Error Handling

### Exception Hierarchy

```typescript
// Use NestJS built-in exceptions for HTTP errors
import {
  BadRequestException, // 400
  UnauthorizedException, // 401
  ForbiddenException, // 403
  NotFoundException, // 404
  ConflictException, // 409
  InternalServerErrorException, // 500
} from '@nestjs/common';

// Custom exceptions for domain errors
export class DomainException extends Error {
  constructor(
    message: string,
    public readonly code: string,
    public readonly traceCode: string,
  ) {
    super(message);
  }
}

export class ExternalApiException extends DomainException {
  constructor(apiName: string, message: string) {
    super(
      `External API error: ${apiName}`,
      'EXTERNAL_API_ERROR',
      `EXT_${apiName.toUpperCase()}_${Date.now()}`,
    );
  }
}
```

### Global Exception Filter

```typescript
@Catch()
export class GlobalExceptionFilter implements ExceptionFilter {
  private readonly logger = new Logger(GlobalExceptionFilter.name);

  catch(exception: unknown, host: ArgumentsHost) {
    const ctx = host.switchToHttp();
    const response = ctx.getResponse<Response>();
    const request = ctx.getRequest<Request>();

    const { status, body } = this.buildErrorResponse(exception, request);

    this.logger.error(
      `${request.method} ${request.url} - ${status}`,
      exception instanceof Error ? exception.stack : undefined,
    );

    response.status(status).json(body);
  }

  private buildErrorResponse(exception: unknown, request: Request) {
    if (exception instanceof HttpException) {
      return {
        status: exception.getStatus(),
        body: {
          message: exception.message,
          errorCode: this.getErrorCode(exception),
          traceCode: this.generateTraceCode(),
          path: request.url,
          timestamp: new Date().toISOString(),
        },
      };
    }

    // Unknown error - log full details, return generic message
    return {
      status: HttpStatus.INTERNAL_SERVER_ERROR,
      body: {
        message: 'Internal server error',
        errorCode: 'INTERNAL_ERROR',
        traceCode: this.generateTraceCode(),
        path: request.url,
        timestamp: new Date().toISOString(),
      },
    };
  }

  private generateTraceCode(): string {
    return `ERR_${Date.now()}_${Math.random().toString(36).substr(2, 6).toUpperCase()}`;
  }
}
```

### TraceCode Pattern

Every error response includes a unique `traceCode` for debugging:

- Format: `<PREFIX>_<TIMESTAMP>_<RANDOM>`
- Example: `ERR_1704631234567_A3F2B1`
- Logged with full error details for correlation

## Testing

### Test File Location

Tests are co-located with source files:

```
src/
└── modules/
    └── user/
        ├── user.service.ts
        └── user.service.spec.ts
```

### Unit Test Structure

```typescript
describe('UserService', () => {
  let userService: UserService;
  let userRepositoryMock: MockProxy<Repository<UserEntity>>;

  beforeEach(async () => {
    userRepositoryMock = mock<Repository<UserEntity>>();

    const module = await Test.createTestingModule({
      providers: [
        UserService,
        {
          provide: getRepositoryToken(UserEntity),
          useValue: userRepositoryMock,
        },
      ],
    }).compile();

    userService = module.get<UserService>(UserService);
  });

  describe('findById', () => {
    it('should return user when found', async () => {
      const userId = 1;
      const userEntity: UserEntity = {
        id: userId,
        name: 'John Doe',
        email: 'john@example.com',
      };
      userRepositoryMock.findOne.mockResolvedValueOnce(userEntity);

      const actual = await userService.findById(userId);

      expect(actual).toStrictEqual(userEntity);
      expect(userRepositoryMock.findOne).toHaveBeenCalledTimes(1);
      expect(userRepositoryMock.findOne).toHaveBeenCalledWith({
        where: { id: userId },
      });
    });

    it('should throw NotFoundException when user not found', async () => {
      const userId = 999;
      userRepositoryMock.findOne.mockResolvedValueOnce(null);

      await expect(userService.findById(userId)).rejects.toThrow(
        new NotFoundException(`User with ID ${userId} not found`),
      );
      expect(userRepositoryMock.findOne).toHaveBeenCalledTimes(1);
    });
  });
});
```

### Testing Conventions

| Convention      | Description                                             |
| --------------- | ------------------------------------------------------- |
| File naming     | `<name>.spec.ts`                                        |
| Top describe    | Class name: `describe('UserService')`                   |
| Second describe | Method name: `describe('findById')`                     |
| Test case       | Should statement: `it('should return user when found')` |
| Mock suffix     | `userRepositoryMock`                                    |
| Result variable | `actual`                                                |

### Testing Rules

1. **One test per logic path** - Test both success and failure cases
2. **Use `mockResolvedValueOnce`** - Over `mockResolvedValue` for strictness
3. **Verify call counts** - `toHaveBeenCalledTimes(1)`
4. **Verify call arguments** - `toHaveBeenCalledWith(...)`
5. **Test only public methods** - Private methods tested through public API
6. **Reset mocks in beforeEach** - Fresh state for each test
7. **Check full error** - `toThrow(new NotFoundException(...))` not just message

### E2E Test Structure

```
tests/
└── e2e/
    ├── _helpers/
    │   ├── api-client.ts
    │   └── test-utils.ts
    └── users/
        ├── users-get.e2e-spec.ts
        └── users-post.e2e-spec.ts
```

## Git Workflow

### Branch Naming

```
<type>/<ticket-id>/<description>

Examples:
feature/NTO-123/user-authentication
fix/NTO-456/login-validation
refactor/NTO-789/user-service
```

### Branch Types

| Type         | Description      | Source  | Target  |
| ------------ | ---------------- | ------- | ------- |
| `feature/*`  | New features     | develop | develop |
| `fix/*`      | Bug fixes        | develop | develop |
| `refactor/*` | Code refactoring | develop | develop |
| `docs/*`     | Documentation    | develop | develop |
| `hotfix/*`   | Production fixes | main    | main    |
| `release/*`  | Release prep     | develop | main    |

### Commit Messages

Follow [Conventional Commits](https://www.conventionalcommits.org/):

```
<type>(<scope>): <description>

Examples:
feat(user): add user authentication
fix(auth): correct token validation
refactor(db): optimize query performance
docs(readme): update installation steps
test(user): add unit tests for UserService
```

### Types

| Type       | Description      |
| ---------- | ---------------- |
| `feat`     | New feature      |
| `fix`      | Bug fix          |
| `refactor` | Code refactoring |
| `docs`     | Documentation    |
| `test`     | Tests            |
| `chore`    | Maintenance      |
| `perf`     | Performance      |

### ESLint Configuration

Use `eslint-config-triple-node` as base with NestJS overrides.

1. **Run before commit** - `npm run lint`
